## Highlights

0.5.20 is a significant feature release. Non-admin users can now self-serve their own Slack/Webex channel grants and bulk-onboard connectors without platform admin intervention. Platform Health grows into a full operational dashboard with per-dependency probes, one-click Slack route repairs, and Webex admin access enabled by default. Multi-replica deployments stop randomly logging users out thanks to a MongoDB-backed token store. New MCP servers default to Streamable HTTP, and two regression fixes restore the Kubernetes port-parsing behavior and docker-compose first-install experience that regressed in the 0.5.x cycle.

## What's New

### Self-Service Integrations
- **Non-admin channel grants** — Slack and Webex integration tabs now return `self_service` mode for non-admin users, letting them grant their own channel or space access without requiring full admin privileges ([#1985](https://github.com/cnoe-io/ai-platform-engineering/pull/1985))
- **Bulk connector onboarding** — the Connector Admin Panel shows discovered connectors with checkboxes, per-row agent assignment, and team slug fields so admins can onboard multiple integrations in a single action ([#1985](https://github.com/cnoe-io/ai-platform-engineering/pull/1985))
- **MCP credential relink on OAuth re-consent** — when a user re-consents to an OAuth provider, pinned `credential_sources` on MCP servers are automatically updated to the new connection so tools keep working without manual re-configuration ([#1985](https://github.com/cnoe-io/ai-platform-engineering/pull/1985))
- **MCP credential fallback** — if a pinned OAuth connection is disconnected, credential resolution now falls back to any active replacement connection the caller owns, preventing tool failures during token rotation ([#1985](https://github.com/cnoe-io/ai-platform-engineering/pull/1985))

### Platform Health & Administration
- **Expanded health probes** — `/api/platform/health` now runs individual dependency probes (MongoDB, Redis, RAG, Keycloak, OpenFGA, AgentGateway, Webex) and surfaces remediation links next to each failing probe. The Platform Health widget in the UI shows per-service status instead of a single aggregate Down/Up badge ([#1992](https://github.com/cnoe-io/ai-platform-engineering/pull/1992))
- **Webex admin API enabled by default** — `WEBEX_ADMIN_API_ENABLED` now defaults to `true` in both Compose and Helm, making Webex space discovery and admin operations available without a manual config change ([#1992](https://github.com/cnoe-io/ai-platform-engineering/pull/1992))
- **One-click Slack route repair** — the Slack/ReBAC diagnostic panel now shows actionable repair buttons for common routing mismatches and supports batch-fixing multiple routes at once ([#1992](https://github.com/cnoe-io/ai-platform-engineering/pull/1992))
- **Changelog filtered to stable releases** — the About dialog changelog only lists stable `x.y.z` releases, omitting dev pre-release entries ([#1992](https://github.com/cnoe-io/ai-platform-engineering/pull/1992))

### MCP Transport
- **Streamable HTTP default** — new MCP servers are now created with Streamable HTTP as the default transport. Existing SSE servers continue to work and can still be edited as SSE ([#1992](https://github.com/cnoe-io/ai-platform-engineering/pull/1992))

## Bug Fixes

- **caipe-ui (multi-replica auth)**: Replace per-pod in-memory token Map with a MongoDB-backed L1+L2 store — pods in multi-replica deployments shared no token state, causing random session logouts when requests landed on different pods. Tokens are now stored in MongoDB with AES-256-GCM encryption (key derived from `NEXTAUTH_SECRET`) and cached in-memory with a 60-second TTL. Single-replica deployments with no `MONGODB_URI` are unaffected ([#1987](https://github.com/cnoe-io/ai-platform-engineering/pull/1987))
- **ui/chart**: Re-add `envPort()` helper for Kubernetes `tcp://` port env vars — Kubernetes auto-injects `{SERVICE}_PORT` as `tcp://host:port` strings; passing these directly to `Number()` returned `NaN`, crashing `/api/platform/health` with HTTP 500 every 30 seconds and filling pod logs with `RangeError: Port should be >= 0 and < 65536`. The fix from #1927 had regressed in 0.5.19 ([#1983](https://github.com/cnoe-io/ai-platform-engineering/pull/1983))
- **docker-compose**: Wire OpenFGA into the first-install UI — fresh OSS installs using the default profiles reached the migrations page but ReBAC migrations failed with `OPENFGA_HTTP is not set`. The published release image tag (not the `-prod` suffix) and `OPENFGA_HTTP`/`OPENFGA_STORE_NAME` defaults are now set correctly in `docker-compose.yaml` ([#1907](https://github.com/cnoe-io/ai-platform-engineering/pull/1907))
- **ui**: Scope `/api/chat/shared` pre-filter to sharing-configured conversations — the shared chat endpoint was returning results for conversations that had not been explicitly shared, exposing unintended chat history
- **mcp**: Route Confluence provider tokens through AgentGateway — Confluence MCP tool calls were bypassing the AgentGateway auth layer and failing with credential errors

## Breaking Changes

No breaking changes. Drop-in upgrade from 0.5.19.

## Upgrade Guide: 0.5.19 → 0.5.20

### Helm Values Changes

#### Default Changed

| Key | Old default | New default | Impact |
|-----|-------------|-------------|--------|
| `caipe-ui.env.WEBEX_ADMIN_API_ENABLED` | `"false"` | `"true"` | Webex admin API is now active out of the box. If your deployment does not have a Webex admin bot token configured and you want to suppress the resulting health warnings, set this key explicitly to `"false"` in your `values.yaml`. |

No keys were renamed, removed, or added as required. All other values are unchanged.

### Upgrade Runbook

#### 1. Review Webex admin API default

If you do not have a Webex admin bot token and want to keep the admin API disabled:

```yaml
caipe-ui:
  env:
    WEBEX_ADMIN_API_ENABLED: "false"
```

Otherwise, no `values.yaml` changes are needed.

#### 2. Upgrade the chart

```bash
helm upgrade ai-platform-engineering \
  oci://ghcr.io/cnoe-io/charts/ai-platform-engineering \
  --version 0.5.20 \
  -f your-values.yaml
```

#### 3. Verify

```bash
kubectl get pods -n <namespace>
# All pods should reach Running state within ~60 s

curl http://<ui-host>/api/platform/health
# Expect HTTP 200 with per-probe statuses
```
