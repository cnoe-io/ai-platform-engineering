## GAP #35 — SA blocked by org-level `supervisor#invoke` gate (SPEC GAP + runtime bug)

**Symptom**: An SA with valid `service_account:<sub> can_use agent:<id>` is 403'd at `POST /api/chat/conversations` (and any `/chat/invoke` path) by `requireRbacPermission(session,'supervisor','invoke')` → `organization:caipe#can_chat`, checked as hardcoded `user:<sub>` (`api-middleware.ts` org-gate path, ~line 825). The SA is not an org member and the create route grants no org/chat capability, so it 403s BEFORE the DA agent-use check (FR-011) ever runs.

**Spec read**:
- **Is the chat/invoke capability specified anywhere?** NO. spec.md FR-010..013, the OpenFGA-tuples table in data-model.md (only `owner_team` + `can_use agent` + `can_call tool`), and the model change (`owner_team`/`can_manage`) contain **no** chat/invoke/org-membership grant for the SA. → **Genuine spec gap.**
- **Is it implied by an FR?** YES. FR-010 ("authenticate external callers… as that service account") and FR-011 ("authorize agent invocations against the SA's granted access — allowing only agents the SA was granted") only have meaning if the SA can actually *reach* the agent-invoke path. The spec's own contracts/service-accounts-api.md (lines 145-152) shows the SA POSTing to `/api/v1/chat/invoke` "the same route browsers/Slack use" and expecting the DA `can_use agent` check to be the gating decision. So a capability sufficient to pass the upstream supervisor gate is **implied/required** by FR-010/011 — the spec just never enumerated the tuple.
- **Decisive spec language re: over-granting**: FR-004 — "A newly created service account MUST have access to **nothing** until scopes are explicitly granted." The spec's Assumptions also state SA access "is enforced by the same authorization checks that govern human callers, evaluated against the service account's identity rather than a user's." Granting **org-membership** (`organization:caipe#member`) to every SA would (a) tension with FR-004's default-deny ethos and (b) potentially confer every member-level org capability (directory search, files, feedback, ai_assist, credential_vault, skill use…), i.e. far more than "use the agents I was granted." That is an over-grant.

**Most spec-consistent fix (spec-fidelity recommendation)**: a **narrow** capability that lets the SA pass the supervisor/`chat_supervisor#can_chat` gate **only**, NOT blanket org-membership. Two defensible shapes (code team to choose mechanism):
  (a) make the supervisor gate SA-aware so a holder of ≥1 `service_account:<sub> can_use agent:*` is permitted to invoke (the agent-use + caller-keyed tool checks remain the real authorization), or
  (b) write a single narrow tuple at create time granting the SA exactly the `chat_supervisor#can_chat` / `organization:caipe#can_chat`-equivalent — and nothing else.
Both honor FR-004 better than org-membership. Whichever is chosen, the supervisor-gate code MUST namespace the subject as `service_account:` (the org-gate currently hardcodes `user:${subject}`, so even an org tuple on the SA wouldn't be checked under the right subject — that hardcoding is itself part of the bug).

**Spec/doc update (recommend YES regardless of code fix)**: add the chat/invoke capability tuple to data-model.md's OpenFGA-tuples table and a one-line FR (or amend FR-010/011 note) in spec.md stating that creating an SA also grants the minimal capability required to invoke the supervisor/chat path, scoped no wider than necessary — so the spec matches reality and future readers don't re-hit this. Flag the FR-004 boundary explicitly (narrow capability ≠ org membership).

**Bottom line**: This is a real spec gap (capability never enumerated) AND a runtime bug (gate hardcodes `user:`). US2 end-to-end cannot pass until it's fixed. FR-010/011 are verified-in-code for the SA-identity/agent-use logic but **runtime-blocked** by this gate.
